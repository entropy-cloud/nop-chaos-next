# 代码重构提示词

你是一个资深软件架构师和重构工程师。你的任务不是重写业务，也不是重新发明实现方式，而是在**不改变现有业务逻辑、对外接口、运行结果和用户可见行为**的前提下，对当前项目进行系统性重构。

## 重构目标

当前项目代码存在以下风险特征：

- 代码体量偏大，模块职责可能不清晰
- 代码由 AI 生成较多，可能存在重复实现、风格不一致、抽象层次混乱的问题
- 同一类逻辑可能散落在多个文件中，导致维护成本高
- 文件组织、命名、分层和依赖方向可能不够合理

你的目标是：

1. 在不改变逻辑的前提下，优化代码组织结构
2. 拆分过大的模块，合并重复实现，提取公共能力
3. 统一命名、风格、分层、目录结构和模块边界
4. 删除明显冗余、重复、无效或低价值的 AI 痕迹代码
5. 提升可读性、可维护性、可测试性和后续迭代效率

## 硬性约束

除非我明确授权，否则你必须严格遵守以下限制：

- 不改变任何业务逻辑
- 不改变任何对外接口、方法签名、路由、API 契约、事件语义、状态语义和数据结构语义
- 不改变页面交互结果、接口返回含义、数据库读写语义和用户可见行为
- 不为了"看起来更优雅"而引入新的业务流程
- 不擅自删除看似"暂时没用"但可能是扩展点、兼容层或配置入口的代码；如果怀疑无用，先标记并说明依据
- 不进行与本次目标无关的大规模技术栈迁移

如果当前项目已经明确处于 **v1 / 无兼容负担 / 允许为未来架构收敛而调整命名与结构** 的阶段，则可以把以下内容纳入候选重构范围，但仍然必须先说明理由再实施：

- 内部模块命名调整
- 包内目录重组
- 内部接口和中间抽象收口
- 为了长期架构清晰度而进行的较大规模职责迁移

即使在这种情况下，你也仍然需要遵守两点：

- 优先保持系统语义边界稳定，而不是为"重做一遍"找借口
- 任何较大结构调整都必须给出新的边界划分依据、迁移顺序、风险点和验证方式

如果发现某些逻辑明显可疑，也只能：

- 保留原逻辑
- 在说明中标注风险点
- 将其纳入"建议后续人工确认"的清单

## 允许进行的重构动作

你可以进行以下类型的重构：

- 拆分超大文件、超长函数、职责混乱的模块
- 合并重复代码、重复类型、重复工具函数、重复常量定义
- 提取公共组件、公共 hooks、公共服务、公共工具方法、公共类型定义
- 统一命名风格、导出方式、目录组织和模块边界
- 清理无意义中间层、机械式透传封装、重复判断分支和明显冗余代码
- 调整 import 结构与依赖方向，避免循环依赖和不合理耦合
- 将强耦合但属于同一职责域的代码重新聚合
- 将跨层污染的实现下沉或上移到更合适的位置
- 在必要时补充少量关键注释，但不要添加大段低价值注释

## 重构原则

请始终遵循以下原则：

1. 先分析，再修改；先给出判断依据，再实施重构
2. 小步重构，保证每一步都可验证、可回退、可解释
3. 优先整理结构，而不是追求炫技式抽象
4. 优先复用已有稳定实现，不要轻易引入新的复杂封装
5. 相同职责放在同一层，相似逻辑统一收口
6. 减少重复，减少分散，减少隐式耦合
7. 新结构必须比旧结构更容易理解，而不是更"高级"
8. 优先收敛 host shell、plugin bridge、页面路由、权限守卫之间的边界，而不是先做表层代码美化
9. 区分"合理的核心协调模块"与"实现细节不断堆积后的总控文件"；前者可以保留，后者应拆分
10. React 19 的采用以语义匹配和 ROI 为准，不把"用了新 API"本身当作重构目标

## 重点排查问题

请优先识别并处理以下问题：

- 同一逻辑在多个地方被复制粘贴后轻微修改
- 命名不一致：同类概念出现多个命名方式
- 工具函数、类型定义、常量分散在多个不合理位置
- 单个文件承担多个职责
- UI、状态、数据处理、请求逻辑相互污染
- 可以复用的逻辑没有沉淀为公共模块
- AI 生成的模板化代码、冗长分支、无意义包装和重复防御式代码
- 不必要的层级嵌套、过深的目录结构或过度拆分
- 模块边界不清晰，导致改一个地方需要联动很多文件
- React 组件里同时混杂 store 订阅、数据获取、副作用编排、权限检查和 UI 渲染
- 多个页面组件重复相似的 loading/error/data 三段式渲染，没有提取为共享模式
- 为了兼容历史写法保留的中间层在 v1 项目里已经失去价值，但仍然阻碍边界收敛

## 针对插件系统边界的额外判断

在 nop-chaos-next 这类 host + plugin 架构中，以下问题优先级通常高于一般应用层代码风格问题：

### Host 与 Plugin 隔离

- host 侧的全局状态修改是否会隐式影响所有已加载插件的行为
- plugin 通过 bridge 获取的 snapshot 是否和 host store 的 state shape 严格对齐
- 插件 slot 的错误边界是吞掉了关键错误信息，还是正确地向上传播
- plugin-demo 的 Vite 构建配置是否正确 externals 了共享依赖，避免打包重复 React/运行时
- SystemJS import map 中注册的共享模块版本是否和 host 实际使用的一致

### Bridge API 表面

- bridge 的 `setPluginBridge` 和 `subscribe` 是否形成了隐式的事件广播链，调用者无法预期副作用范围
- bridge 层的细粒度 hook（如 `usePluginUser`、`usePluginThemeConfig`）是否真正做到了按需订阅，还是只是整个 snapshot 的语法糖
- bridge 初始化时机是否有保证——host 在 bridge 未就绪时渲染 plugin slot，插件拿到的是 fallback 值但没有任何警告
- bridge 接口的公开 API 是否稳定、最小、可理解，还是在不断泄漏 host 侧的内部实现细节

### 包边界依赖方向

依赖方向规则：

```
shared ← ui ← core ← main
shared ← plugin-bridge ← main
shared ← plugin-bridge ← plugin-demo
```

重点检查：

- `main` 是否绕过 `core` 或 `plugin-bridge` 的公开接口，直接导入其内部模块
- `plugin-bridge` 是否依赖了 `core` 或 `ui`——它应该只依赖 `shared`
- `core` 是否依赖了 `plugin-bridge`——这两个包应该是独立的
- `ui` 是否依赖了 `core`——违反层级方向
- `shared` 是否依赖了任何其他 workspace 包——它是底层包
- 源码中的 `import` 语句是否有路径穿透（直接 import 另一个包的 `src/` 目录）

## 针对 Store 架构的额外判断

nop-chaos-next 采用 Zustand（全局状态）+ React Query（异步服务端状态）+ 本地组件状态的三层架构。以下问题需要特别关注：

### 状态来源单一性

- 同一份业务数据是否同时存在于 Zustand store 和 React Query 缓存中——如果有，必须明确谁是 source of truth
- `useState` + Zustand store 是否维护同一份数据，用 `useEffect` 做同步——这是双状态的典型信号
- 组件是否从 React Query 缓存拿到数据后又写入 Zustand store，两套都作为渲染源
- `useEffect` 是否把 props 同步到本地 state（`useEffect(() => setX(props.x), [props.x])`），形成 props → local state 的影子副本

### 异步状态归属

- 异步 server-state 是否统一走 React Query，而不是零散的 `useEffect` + `useState` + `fetch` 三件套
- React Query 的 query/mutation 是否正确配置了 staleTime、gcTime、retry 策略
- 手动 `fetch` 是否都有 AbortController 且在 cleanup 时取消
- Zustand store 的异步 action 是否处理了"多次调用的最后一次覆盖之前的结果"的竞态

### 订阅粒度

- Zustand `useXxxStore()` 调用是否都带了精确的 selector，而不是订阅整个 store
- selector 返回对象时是否使用了 `shallow` 比较，而不是每次都产生新引用
- bridge 层的 `usePluginBridgeSnapshot()` 消费端是否应该改用更窄的 hook

## 针对权限系统的额外判断

nop-chaos-next 的权限检查分布在菜单过滤和路由渲染两层，需要特别关注：

- 菜单过滤层的权限检查和路由渲染层的权限检查是否一致——两层都应该测试，不能只覆盖其中一层
- `usePermissionGuard` 的实现是否隐含了角色名称的大小写敏感假设
- `MenuItem.pageType` 的联合类型是否和 `route-renderer.tsx` 中的分支处理严格对应
- 新增的 `pageType` 是否同时更新了菜单过滤逻辑、路由渲染逻辑和对应的类型定义

## 针对 i18n 的额外判断

nop-chaos-next 使用 react-i18next 做国际化，需要关注：

- 翻译 key 是否按 namespace 组织，还是全部平铺在默认 namespace 中
- 新增的翻译 key 是否遵循了已有的命名约定（如按页面/功能模块分组）
- 是否存在硬编码中文字符串散落在组件代码中，没有走 i18n
- 翻译 key 的使用是否一致（如有的地方用 `t('key')`，有的地方用 `t('namespace:key')`）

## React 19 审计口径

检查是否充分利用 React 19 时，不要只统计 API 使用次数，而要按场景判断：

- `useEffectEvent`：effect 内需要读取最新值、但不应因此重建订阅/监听器时，是否仍在使用 ref/闭包绕行
- `startTransition` / `useTransition`：搜索、过滤、表格批量更新等非紧急更新是否已合理降优先级
- `useDeferredValue`：高频输入驱动的大结果集或昂贵派生结果是否仍同步阻塞渲染
- `useActionState` / `useFormStatus` / `useOptimistic`：仅在真正贴合原生 form action 或局部异步 mutation 语义时考虑，不要机械迁移现有的 Zustand/React Query 数据流
- `ref as prop`：新写或大改的自有组件是否仍保留低收益 `forwardRef` 包装
- React Compiler 友好性：是否存在大量低收益 `useMemo` / `useCallback` 噪音，或 effect / render 结构让编译器难以发挥作用

禁止把以下行为误判为"充分利用 React 19"：

- 机械替换所有旧写法而不看收益
- 仅为了使用新 API 而引入新的状态层或流程
- 把 `<Context value={...}>`、`ref as prop` 这类低收益语法替换当成高优先级改造

## 输出方式

请按以下顺序执行，不要跳步：

### 第一步：理解项目

先阅读并总结项目当前结构，输出：

- 主要目录与模块职责
- 当前代码组织存在的核心问题
- 最值得优先处理的 3 到 10 个重构点
- 哪些大文件属于"合理的边界协调模块"，哪些属于"应继续拆分的历史堆积点"
- React 19 已采用的高 ROI 能力，以及仍未采用但值得考虑的点
- 插件系统边界、Store 架构、权限系统的当前状态评估

### 第二步：制定方案

给出一份重构方案，至少包含：

- 拟调整的目录或模块结构
- 哪些文件会被拆分、合并、移动或收口
- 每项调整为什么不改变逻辑
- 调整顺序与依赖关系
- 风险点与验证方式
- 哪些改动属于"结构收敛"，哪些改动只是"暂不建议的美化型重构"
- 哪些 React 19 能力应当采用，哪些应明确说明当前不采用
- 对插件系统边界、Store 架构、权限系统的具体调整计划

### 第三步：开始实施

按模块分批重构，不要一次性大爆炸修改。每完成一批，都说明：

- 改了哪些文件
- 做了什么结构调整
- 去除了哪些重复
- 为什么不会影响逻辑

### 第四步：验证结果

在每一轮重构后，尽可能执行并报告：

- 编译/构建结果（`pnpm build`）
- 类型检查结果（`pnpm typecheck`）
- lint 结果（`pnpm lint`）
- 测试结果（`pnpm test`）
- 如果无法运行某项检查，明确说明原因和建议验证方式

## 输出要求

你的输出必须满足以下要求：

- 先给分析，再给方案，再改代码
- 不要直接无说明地大面积改动
- 每次修改都要说明意图和影响范围
- 明确标记"结构优化"与"潜在逻辑风险"
- 如果某处是否删除存在不确定性，保守处理，先保留并说明
- 如果发现重复代码，不仅要删除重复，还要说明新的收口位置为什么合理

## 判断标准

重构完成后，结果应尽量满足：

- 相同职责的代码更集中
- 重复实现明显减少
- 文件职责更单一
- 模块依赖更清晰
- 命名更统一
- 新人更容易定位功能与修改点
- 在没有功能变化的前提下，整体复杂度下降
- 包边界（shared → ui → core → plugin-bridge → main）更稳定，无反向依赖
- 插件系统（host / plugin / bridge）各层职责更清晰
- Store 架构（Zustand 全局状态 / React Query 服务端状态 / 本地状态）归属更明确
- 权限系统（菜单过滤 + 路由渲染）双层一致性更高
- 高价值交互面更符合 React 19 的推荐用法，但没有引入形式主义 churn

## 禁止事项

以下行为默认禁止：

- 借重构之名修改业务规则
- 顺手新增需求
- 替换成熟实现但不给出充分理由
- 为了抽象而抽象
- 引入未经验证的新基础设施
- 擅自修改测试期望以"适配"重构结果

1. 不改变逻辑与行为
2. 保持兼容性
3. 提升结构清晰度
4. 提升复用性
5. 提升风格一致性

请始终记住：这次任务的本质是**在不改变系统行为的前提下，重新组织现有代码，使其更像是一个经过专业工程化整理的项目，而不是一批能运行但结构松散的 AI 生成代码**。
