# 废弃特性清理提示词

清理工程中已确认废弃的特性、字段、函数、类型和兼容路径。

## 适用条件

只有同时满足以下两条件的项才可清理：

1. **文档明确标记废弃**：在 `docs/` 下的 design、plans、logs、bugs 中有明确文字说明该项应废弃、待移除、或已被替代。
2. **代码核实确认可移除**：grep 全仓库确认无活跃消费者，或消费者已有替代路径。

缺少任一条件的，只能标记 `@deprecated` 并记录评估结论，不能删除。

## 执行步骤

### 1. 建立废弃候选清单

从文档侧出发，扫描以下来源：

- `docs/plans/` 中 `Follow-up` 或 `Non-Goals` 里明确提到"下个版本移除"或"待清理"的项
- `docs/design/` 中标注 `@deprecated`、`legacy`、`compat`、`将被移除` 的接口或字段
- `docs/bugs/` 中修复后遗留的兼容路径，且 bug 已关闭
- 源码中标注 `@deprecated` JSDoc 注释的导出，但前提是 docs 中也有对应记录

对每个候选项，记录：标记位置（文档路径+段落）、废弃原因、预期替代方案。

### 2. 逐项代码核实

对每个候选项：

1. grep 全仓库搜索所有引用点（导入、调用、测试、schema）
2. 区分：
   - **活跃引用**：生产代码中的真实调用 -> 不能删，需先迁移消费者
   - **测试引用**：仅测试中使用 -> 测试应一并清理或改为测试替代方案
   - **自身声明**：被废弃项本身的定义 -> 可删除
   - **零引用**：无任何引用 -> 可直接删除
3. 若存在活跃引用但文档说"已迁移"，核实消费者是否确实有替代路径。若替代路径未完成，该项不可删除，记录为"迁移未完成"。

### 3. 执行清理

按以下优先级分组执行：

**第一批：零引用的死代码**

- 直接删除定义和对应的文档段落

**第二批：仅测试引用**

- 删除定义，同步清理或删除相关测试用例

**第三批：有活跃引用但替代路径已完成**

- 先迁移消费者到替代方案
- 确认迁移后无引用
- 再删除定义和文档段落

每批完成后立即运行 `pnpm typecheck && pnpm build && pnpm lint && pnpm test`。

### 4. 收尾

- 更新 `docs/logs/` 记录清理内容
- 若某个废弃项经核实后发现不应废弃（文档判断有误），在 docs 中更新结论而非删除代码
- 汇总输出：已清理项清单、保留但标记 @deprecated 的项、判定不应废弃的项

## 安全红线

- 不删除任何未经文档+代码双重确认的项
- 不删除仅有 `@deprecated` 注释但 docs 中无记录的项（可能还在兼容期）
- 不修改废弃项的语义或行为（要么原样保留，要么完整删除）
- 不以"看起来没人用"作为删除依据，必须 grep 验证
