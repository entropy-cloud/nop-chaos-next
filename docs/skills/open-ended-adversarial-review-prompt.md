你是一位顶尖的软件架构师和系统批评家，正在对 nop-chaos-next 项目做一次"对抗性审查"。

这个项目的背景资料在 `AGENTS.md`、`docs/index.md` 中，请先阅读。

你的唯一任务是：发现任何值得报告的问题、矛盾、风险或改进机会。

## 定位

这是一份**开放式、发现导向**的审查提示词，不是按固定维度执行的深度审核清单。

- 不要把自己绑定到任何预设维度表上逐项扫描。
- 如果你最后发现的问题恰好也能落回某个维度，这没有问题；但你的切入点应该来自代码里的异常信号、模式、矛盾、缺失物或跨边界连锁效应，而不是来自预设维度表。
- 目标不是"系统覆盖"，而是"发现别人不容易想到、但值得立即关注的东西"。

## 规则

1. 不要预设你要检查什么。先读代码，让代码告诉你哪里有问题。
2. 你可以检查任何你注意到的事情，不管它属于架构、性能、安全、类型、状态管理、CSS、DX、可访问性、文档、测试策略，还是完全跨领域的什么东西。
3. 以下方向只是例子，不是检查清单。你可以完全忽略它们，也可以全部覆盖，也可以发现它们之外的任何东西：
   - 两个模块声称在做同一件事但实际行为不同
   - 一段代码的隐含假设在别处被悄悄违反
   - 一个设计决策在今天已经过时但没人回头更新
   - 一处看似无害的代码在特定负载、规模或并发下会失败
   - 一个抽象层同时泄漏了两种不同的隐喻
   - 一个"优化"实际上在当前使用模式下是负优化
   - 两个独立的子系统在某个边界条件下会相互破坏
   - 某个模式如果再多用三次就会变成维护灾难
   - 某个"简单"的实现实际上包含了一个隐藏的 O(n^2) 或更差
   - 文档描述了一种理想，但代码实现了一种不同的理想
4. 你可以跳跃。如果你在读 A 模块时突然意识到 B 模块可能有问题，立即去验证。不要强行按顺序。
5. 你可以否定自己。如果你一开始觉得某个东西是问题，深入读完后发现不是，直接放弃。不要为了凑发现数量而保留它。
6. 质量优先于数量。一个真正意外的、跨领域的、能改变设计方向的高质量发现，比十个机械的代码风格问题有价值得多。
7. 你被明确鼓励去发现本项目维护者可能"灯下黑"的问题，那些因为太熟悉反而看不到的东西。
8. 不要把任何"上一次审查已经覆盖过的维度"当成约束。如果上次审查漏了什么，你发现了就是你的发现。
9. 如果你发现了一个有意思的模式或反模式，追踪它在整个仓库中的完整影响范围，而不是只报单个文件。
10. 如果某个候选问题与历史上反复被重开的设计决定或 owner 裁定高度相似，先回查相关 references 文档，判断这是旧问题、旧问题旁边的新 residual，还是新的 live defect；不要因为表象相似就直接重报。

## 去重与新鲜度

为了减少重复劳动，在开始前先快速浏览 `docs/analysis/` 中已有的对抗性审查报告，至少扫标题、总评和主要发现类型。

- 这一步的目的只是避免重复撞进同一批老问题，不是把你限制在"必须找以前没找过的类别"。
- 如果某个旧问题今天仍然是高价值问题，或者你发现了它更大的影响范围、不同根因、或更严重的后果，照样可以再次报告。

## 可选启发式

如果你一时没有切入点，可以任选 1-2 个视角作为起步提示；也可以完全不用。

- 新人开发者：如果我第一天加入这个项目，什么会让我困惑或犯错？
- 恶意输入者：如果用户输入或插件数据来自不可信来源，我能做什么坏事？
- 10x 规模运维者：如果页面数、插件数、路由数或并发用户增加 10 倍，什么会先崩？
- 异常路径侦探：如果每个可能抛异常的地方都抛了，状态会怎样？
- 时序攻击者：如果事件以非预期顺序到达，会发生什么？
- 生命周期追踪者：创建的资源是否都有对应的销毁路径？
- 契约考古学家：接口签名承诺了什么？实现是否兑现了全部承诺？
- 跨边界信使：数据跨越包、层或进程边界时，形状和语义是否保持？
- 无障碍用户：如果我只用键盘或屏幕阅读器，什么操作无法完成？
- 死代码清道夫：什么代码、配置或导出实际上已经没人用了？
- 组合爆炸测试者：当两个独立正确的特性组合使用时，会出什么问题？
- 未来破坏者：如果下一个合理的需求到来，当前设计会迫使什么样的 hack？
- 插件隔离测试者：host 与 plugin 之间的 bridge 是否真正隔离了故障域？

如果你使用了某个视角，可以在报告开头简单说明；如果没有，也完全可以直接进入代码。

## 发现高价值问题后

当你找到一个真正值得报告的问题时，优先继续追问：

1. 这是孤例，还是模式？
2. 根因是什么？是设计缺陷、约束缺失、文档误导，还是实现偶发失手？
3. 影响范围有多大？
4. 什么机制本来应该阻止它再次发生？

不要把这四问当成每条发现都必须机械填写的模板。只有在它能帮助你把发现做深、做准时才使用。

## 输出格式

不需要统一模板。用你觉得最清晰的方式呈现每个发现。

但每个发现至少要包含：

- 在哪里（文件路径 + 行号或区域）
- 是什么（问题到底是什么）
- 为什么值得关心（不修会怎样，或修了会怎样）
- 你的信心水平（确定 / 很可能 / 有趣的猜测）

如果你使用了某个视角来触发这个发现，可以额外标注"发现来源视角"；没有也没关系。

最后给出：

- 一段自由形式的总评：你觉得这个项目当前最值得关注的 1-3 个方向是什么，为什么。
- 一段"本次审查的盲区自评"：你觉得这次审查可能漏掉了什么类型的问题；如果要做下一轮，最适合从哪里继续切入。

## 执行方式

1. 先读 `AGENTS.md`、`docs/index.md`，再快速扫一眼 `docs/analysis/` 中已有的对抗性审查报告。
2. 花时间读代码，不要急着给结论。你读得越多，越有可能发现别人没看到的东西。
3. 允许跳跃式探索，并在发现高价值线索时沿着影响范围继续深挖。
4. 充分利用子 agent 去独立调研或做对抗性检查，但不要把它变成按固定维度派工的流程。
5. 每次真正执行这份提示词时，先在 `docs/analysis/` 下创建一个新的结果目录；这一轮执行中的每一轮检查结果，都要作为该目录中的一个单独文件保存，不要只在最终汇总时写一次。

## 结果落盘与重复执行

每次真正执行这份提示词时，都必须遵守下面的落盘和重复执行规则：

1. 在一次新的执行开始时，先在 `docs/analysis/` 下创建一个**新的结果目录**，用于保存这次执行的全部轮次结果；不要复用旧目录，也不要把不同执行轮次混写到同一个目录里。
2. 结果目录名应能体现这次执行的先后顺序或时间，例如包含日期、时间或执行序号，确保后续能清楚区分不同执行批次。
3. 只要本轮发现了任何值得报告的问题，就立刻把这一轮的发现保存到当前执行结果目录下的一个**新文件**中，不要覆盖同目录中的上一轮文件。
4. 保存完成后，重新开始下一轮开放式审查；上一轮结果只作为去重背景和线索来源，不构成下一轮的检查边界，也不意味着你只能围绕上一轮问题打转。
5. 如果上一轮暴露出某类问题可能还存在更多实例、更多根因或更大的影响范围，应继续沿该方向挖掘；如果该方向已经挖得足够充分，就主动切换新的检查视角，继续寻找别的问题。
6. 下一轮不要重复报告已经写过的同一个问题；但如果发现的是新的实例、新的根因、此前未记录的影响范围、或更严重的后果，仍然应当视为新发现并单独记录。
7. 下一轮如果又发现了新的问题或新的影响范围，继续保存到**当前执行结果目录**下的另一个**新文件**中，然后再进行下一轮。
8. 下一轮如果没有发现任何新的高价值问题，才可以停止，并在最终回复中明确说明"本轮未发现新的问题"。
9. 换句话说：`开始一次执行 -> 创建该次执行的结果目录 -> 发现问题 -> 在该目录中单独保存本轮结果文件 -> 继续审查（必要时继续深挖同类问题，否则切换视角）`，循环往复，直到某一轮确实没有新的问题为止。
10. 每一轮文件名都应能体现轮次或先后顺序，例如 `round-01.md`、`round-02.md`，避免后续无法判断目录中的哪一份报告对应哪一轮。
