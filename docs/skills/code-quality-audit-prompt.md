# 通用代码质量审计提示词

## 目标

用于对任意代码仓库执行一次**以真实实现质量为中心**的代码质量审计。

这份提示词不把 lint、覆盖率、规范遵循率当成最终结论，只把它们当成辅助线索。审计重点是：

1. 当前实现是否正确表达了系统意图。
2. 复杂度是否被有效组织，而不是任意堆积。
3. 测试是否真正约束高风险行为，而不是只覆盖 happy path。
4. 自动化工具是否覆盖了最值得自动化的风险。

## 使用原则

1. 先读代码，再下结论。
2. 以 live code 为准，不以历史计划、口头约定、旧日志、理想化架构图为准。
3. 不把"我个人更喜欢另一种写法"当成问题。
4. 不把动态边界、orchestrator 模式、框架核心热点、桥接层天然复杂度机械判成坏味道。
5. 每个发现必须有：文件路径、行号范围、问题实质、风险、修复方向。
6. 区分"必要复杂度"和"失控复杂度"。
7. 区分"自动化可发现的问题"和"必须靠读源码才能发现的问题"。

## 审计维度

### 1. 架构与边界

- 包边界、模块边界、依赖方向是否清晰。
- 是否存在绕开公开契约直接依赖内部实现的情况。
- 关键 contract 是否集中、稳定、可追踪。
- 是否存在单一职责明显失守的热点文件或热点模块。

### 2. 核心实现质量

- 核心执行路径是否逻辑清晰、状态来源单一、边界收口明确。
- 复杂逻辑是被拆解成可理解模块，还是只是横向分散到很多文件。
- 是否存在真实的竞态、生命周期归属错误、双事实来源、错误传播缺口。
- 是否存在"文档很好，但实现实际松散"的情况。

### 3. 类型与契约质量

- 类型系统是否帮助表达约束，而不是大量 `any`、宽泛对象和无意义包装。
- 动态边界上的宽类型是否合理。
- 公共接口是否稳定、一致、可理解。
- 是否存在概念重复：两个类型、两个字段、两套命名表达同一件事。

### 4. React / UI / 宿主-插件桥接质量

- 渲染层是否承担了本不该由它承担的业务状态。
- 订阅粒度、依赖追踪、上下文使用是否克制。
- UI 组件是否只停留在视觉层，还是背负了不该背的领域逻辑。
- 样式体系是否与运行时契约一致，而不是隐式固化行为。
- 插件桥接层是否正确隔离了 host 与 plugin 的故障域。

### 5. 测试有效性

- 是否覆盖真实高风险行为：竞态、边界、生命周期、错误路径、跨层交互。
- 测试是否能在失败时快速定位原因。
- 是否存在过胖测试文件、过重 setup、用例边界模糊的问题。
- 是否存在明显未被测试约束的核心热点。

### 6. 可维护性与演进风险

- 热点文件是否持续吸收新职责。
- 是否存在长期会拖慢演进的结构性问题，而非局部代码风格问题。
- 是否有"现在能跑，但后续每次改动都危险"的模块。

### 7. 自动化覆盖面

- 现有 lint、typecheck、build、test、repo checks 实际能发现什么。
- 哪些高 ROI 风险应补自动化。
- 哪些问题不适合依赖自动化，而应通过人工审计或架构 review 发现。

## 审计步骤

1. 先识别仓库核心包、核心入口、核心运行路径。
2. 选取最能代表系统本质的源码，而不是只读 barrel 或 README。
3. 抽读关键实现文件，建立对真实架构的理解。
4. 抽读关键测试，判断测试是否真正约束复杂行为。
5. 再看自动化脚本、lint、check、coverage、CI 规则，把它们当作辅助证据。
6. 最后输出结论：先给发现，再给判断，再给自动化建议。

## 输出要求

### 发现优先

按严重度排序输出发现。每个发现必须包含：

1. 严重度：`P0` / `P1` / `P2` / `P3`
2. 类别：如架构边界、运行时逻辑、测试质量、重复实现、复杂度热点、自动化缺口
3. 位置：文件路径 + 行号范围
4. 现状：一句话说明问题是什么
5. 为什么重要：说明真实风险
6. 修复方向：一句话说明建议

### 严重度标准

- `P0`: 当前已构成错误行为、安全风险、核心数据损坏风险、或硬性边界违约。
- `P1`: 高概率回归、核心契约漂移、热点设计错误、或关键测试缺失。
- `P2`: 真实维护成本问题、局部重复、局部抽象失衡、或热点持续膨胀风险。
- `P3`: 观察项、轻度组织问题、或暂不值得立即改动的问题。

## 自动化工具建议模板

在报告中单列"自动化工具"章节，区分三类：

1. 已有且有效：当前仓库已经具备，应继续保留。
2. 建议补充：高 ROI、适合当前仓库结构的自动化工具。
3. 不应过度依赖：容易制造形式主义指标，但无法代表真实实现质量的工具。

建议优先考虑：

- `dependency-cruiser`：边界、依赖方向、跨层违规
- `knip`：未使用文件、导出、依赖
- `Stryker`：核心逻辑包的变异测试
- `CodeQL` / `Semgrep`：高风险模式扫描
- repo-specific checks：文件大小、产物泄漏、旧 API、i18n、公共表面等

## 可直接复用的提示词正文

```text
请对当前仓库执行一次"真实实现质量导向"的代码质量审计。

要求：
1. 不要只看 lint、coverage、规范遵循率。
2. 必须阅读核心源码与关键测试，判断真实实现质量。
3. 先找出最关键的架构/运行时/桥接热点文件，再给结论。
4. 区分必要复杂度与失控复杂度。
5. 区分自动化可发现的问题与必须人工读源码才能发现的问题。
6. 不把个人风格偏好当成问题，不机械追求"抽象更多"或"文件更碎"。
7. 对动态边界、框架热点、orchestrator 模式保持克制，先判断它是否合理。

输出格式：
1. Findings
   - 按严重度排序列出问题。
   - 每条包含：严重度、类别、文件路径、行号范围、问题现状、真实风险、修复方向。
2. Overall Assessment
   - 给出对仓库整体代码质量的客观判断。
   - 说明哪些部分质量高，为什么高；哪些部分风险高，为什么风险高。
3. Automation
   - 列出已有有效工具。
   - 列出建议补充的工具，并说明它们解决什么问题。
   - 列出不应过度依赖的指标或工具。
4. Evidence
   - 引用实际源码和测试文件，不要只引用入口文件或 README。

结论必须以当前 live code 为准，不以历史计划、旧日志或假设中的理想架构为准。
```
