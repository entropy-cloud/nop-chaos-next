# 下一代前端低代码框架吸引子发现提示词

> **定位**: 这不是一个让 AI 直接"发明新架构"的提示词，而是一个强制 AI 暴露旧吸引子前提、枚举替代切分、比较主流盆地、并输出可验证新吸引子候选的工作流提示词。
> **前提**: 当前主流 AI 受训练分布和对齐偏置影响，默认会朝它见过的平均方案收敛。不要把它当成原创架构来源，而要把它当成高带宽的前提暴露器、替代结构枚举器、反例生成器和收敛验证器。
> **适用对象**: 下一代前端低代码框架、Schema 驱动 UI/runtime、表单/动作/作用域/数据域/宿主能力等核心抽象重构。本提示词是架构无关的通用工具，不包含任何特定框架的术语或结构假设。

---

## 核心术语

为避免脱离上下文后误读，本提示词中的几个词采用下面的专门定义：

- **吸引子**：不是"最佳答案"，而是一组会持续约束系统演化方向的稳定结构。只要系统在它附近展开，很多局部实现最终都会回到同一类概念切分、边界组织和术语系统中。
- **旧吸引子**：当前系统和主流框架默认回流到的结构盆地。它通常不是显式写下来的设计图，而是一组已经被训练语料、现有代码、团队习惯和主流框架反复强化的默认前提。
- **新吸引子**：能够重新定义问题空间的结构语言。它不是旧方案的局部优化、重命名或重新组合，而是会改变核心边界、最小术语系统、评价函数或实现收敛方向的候选结构。
- **主流盆地**：当前训练语料和公共知识中已经高度稳定的方案集合，例如 AMIS、Formily、React Hook Form、JSON Schema Form、BPMN-like action graph、Retool、Appsmith 等的平均路径。AI 很容易回流到这里。
- **发现新吸引子**：不是要求 AI 凭空发明架构，而是要求它帮助暴露旧吸引子的隐含前提，枚举替代切分，并判断某个候选是否真的重写了问题空间。

判别规则：

1. 如果一个候选只是把主流方案重命名、轻度重组装或局部优化，它不是新吸引子。
2. 只有当它重新定义了核心边界、最小术语系统或评价函数时，它才有资格被保留为"新吸引子候选"。
3. 本提示词的目标不是寻找"最佳方案"，而是判断当前系统是否被旧吸引子困住，以及是否存在值得验证的新吸引子。
4. 如果一个候选主要只是把现有系统中已经存在的能力换一组新术语重新表述，它更可能是"重命名收束"，不是新吸引子。
5. 如果一个候选把本来属于 host / application / backend 层的问题上推到 runtime core，却没有证明这种上推带来更稳定的 authoring 或执行边界，它不能因为"更硬"就被当作新吸引子。

---

## 什么时候使用

当你怀疑当前方案只是沿着主流框架的平均路径在演化，而不是在形成真正新的结构语言时，使用本提示词。

尤其适用于以下场景：

- 你已经意识到当前方案"能做出来"，但概念边界仍然脏
- 你怀疑某几个概念被混在一起，但还说不清应该怎么拆
- 你需要从 AMIS / Formily / React Hook Form / JSON Schema form / BPMN-like action graph 等主流盆地里跳出来
- 你想验证某个新方向到底是"换个说法的旧方案"，还是"真正的新吸引子"

---

## 不要让 AI 做什么

这个提示词明确禁止以下行为：

1. 不要让 AI 直接给"最佳架构方案"。
2. 不要让 AI 只做主流方案综述。
3. 不要接受"更灵活""更优雅""更通用"这种空泛评价。
4. 不要把"与现有主流方案略有不同"误判为新吸引子。
5. 不要让 AI 只给实现建议，而跳过概念切分与判定标准。

你要逼 AI 先回答：

- 旧吸引子默认了什么？
- 哪些概念被混在一起了？
- 为什么这些混合会持续制造复杂度？
- 哪一种新切分能够重写问题空间，而不是仅仅优化旧答案？

---

## 使用前输入

在使用主提示词前，先准备这 7 类输入：

1. **当前系统基线**
   - 当前前端低代码框架的核心抽象
   - 当前运行时/编译期/渲染期边界
   - 当前已知的复杂点、脏点、重复模式

2. **上游/主流盆地样本**
   - 例如 AMIS、Formily、React Hook Form、JSON Schema Form、BPMN/action graph 等
   - 不要求完整调研，但至少列出当前系统最像哪些主流路径

3. **当前不满意的症状**
   - 哪些能力经常纠缠在一起
   - 哪些 API 虽然能用，但语义不纯
   - 哪些实现成本来自概念混合，而不是工程细节

4. **你已经怀疑的切分方向**
   - 例如 data vs capability
   - 例如 lexical vs dynamic scope
   - 例如 compile-time lowering vs runtime orchestration
   - 例如 declarative schema vs imperative API

5. **判定目标**
   - 你希望得到的是新的什么：新作用域模型？新 action 语义？新数据 owner 模型？新宿主集成方式？

6. **作者面与分层现实**
   - 当前 authoring surface 是什么：JSON / DSL / visual model / mixed
   - 当前系统的 `name / id / ref`、事件入口、action surface 是如何定位的
   - 当前系统哪些能力属于 authoring 层，哪些属于 compiler / IR / runtime / host / backend

7. **迁移与兼容现实**
   - 当前系统已有多少作者面、JSON 习惯、组件契约、调试工具、编译边界不能轻易推翻
   - 你希望得到的是"全新底座"还是"可渐进迁移的新中心"

---

## 输出契约

AI 的输出必须严格包含以下 13 个部分，缺一不可：

1. **当前吸引子识别**
2. **混合概念清单**
3. **替代切分矩阵**（至少 4 套不同切分方案）
4. **主流盆地回流检测**
5. **新吸引子候选**（最多 2 个）
6. **反例与失败模式**（每个候选至少 3 个）
7. **最小可执行语言**
8. **收敛验证计划**
9. **层级归属审计**
10. **现有系统对应关系审计**
11. **作者面审计**
12. **编译期 / 运行期现实检查**
13. **兼容与迁移审计**

## 主提示词模板

把下面整段作为主提示词使用；将方括号内容替换为你的具体问题域。

```text
你是一位"下一代前端低代码框架架构研究员"，但你的任务不是直接给出一个看起来漂亮的架构答案，而是帮助我发现：当前系统是否被困在主流方案的平均盆地里，以及是否存在值得验证的新吸引子。

你必须遵守以下规则：

1. 不要直接给"最佳方案"。
2. 不要只总结 AMIS / Formily / JSON Schema Form / React Hook Form 等主流路径。
3. 不要把"略微不同的主流方案变体"误报为新吸引子。
4. 你的目标不是优化现有解，而是暴露旧吸引子的隐含前提，并寻找是否存在重新定义问题空间的切分方式。
5. 如果你认为某个候选方案本质上仍然回流到主流盆地，必须明确指出，不允许为了"显得有创意"而保留它。
6. 当前主流 AI 受训练分布和对齐偏置影响，会朝平均方案收敛。你必须主动抵抗这种回流倾向。

[在这里列出你观察到的复杂度、耦合、语义污染、难以维护的问题。]

你必须按以下结构输出：

## 1. 当前吸引子识别

## 4. 主流盆地回流检测

## 6. 反例与失败模式
## 7. 最小可执行语言
## 8. 收敛验证计划

## 10. 现有系统对应关系审计
```

## 推荐追问模板

如果主提示词跑完后结果仍然偏主流、偏保守，继续追问：

### 追问 1：逼出隐藏前提

```text
不要继续给方案。先只回答：当前方案默认了哪些前提，而这些前提其实并不是必然成立的？
请至少列 7 条，并说明每条前提一旦被取消，问题空间会如何改变。
```

### 追问 2：逼出错误切分

```text
如果我坚持沿用当前主流方案的概念切分，未来 2 年最可能反复出现的结构性复杂度会是什么？
不要讲实现细节 bug，只讲概念边界层面的复杂度复发机制。
```

### 追问 3：逼出反主流候选

```text
请只保留那些"明显不属于主流框架平均路径"的候选切分。
如果某个方案与主流路径高度相似，直接删除，不要保留。
```

### 追问 4：逼出新术语

```text
假设这个新吸引子成立，请不要讲实现，先给出它必须引入的最小新术语系统。
如果没有新术语，它大概率不是新吸引子，而只是旧方案改名。
```

### 追问 5：逼出失败条件

```text
请站在反对者立场，证明这个候选吸引子为什么可能是错的。
如果它错了，最先暴露错误的 3 个场景是什么？
```

---

## 评审标准

一个"新吸引子候选"至少要满足下面 5 条中的 4 条：

1. **不是旧方案换词重说**
2. **重新切分了至少一个核心边界**
3. **能解释当前系统里反复出现的复杂度来源**
4. **能导出最小可执行语言，而不只是理念口号**
5. **能设计出最小验证切片，而不是只能停留在抽象层**

新增否决条件：

1. 如果它主要只是把现有系统能力重命名，不得保留
2. 如果它缺少自然作者面，只存在 IR/协议层，不得直接当作候选架构
3. 如果它把 application / backend / host 问题错上推到 runtime core，不得因"更强约束"而加分
4. 如果它只在文档中宣称 compile/runtime separation，但没有说明哪些语义真正被编译固化，不得保留为成熟候选
5. 如果它需要推翻现有 authoring surface，却没有给出自然、低摩擦、可渐进迁移的作者面，不得保留为成熟候选

---

## 使用提醒

- 不要期待一次就发现正确答案。
- 这个提示词的目标不是"生成新架构"，而是"更快识别旧架构为什么不对，以及新架构候选是否值得验证"。
- 真正的新吸引子仍然要由人来裁决；AI 的价值在于加速暴露前提、枚举候选、制造反例、逼近可执行结构。
