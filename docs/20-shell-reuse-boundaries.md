# 主框架复用的扩展边界

> 本文档说明：当主框架要复用于多个应用时，哪些能力应继续通过 router + page component 注册来替换，哪些能力应抽成 shell contribution 配置。目标不是减少扩展能力，而是避免“只想换品牌，却必须整页重写”。

---

## 1. 核心判断

主框架复用时，建议把扩展点分成两层：

- 页面级扩展：通过 router 切换、按 page component 注册替换
- 壳层级扩展：通过 contribution 注入全局配置

一句话原则：

- 结构差异大，直接换 page component
- 只是品牌、入口文案、全局壳信息不同，优先走 contribution

这样可以同时满足两类场景：

- 业务应用需要完全不同的登录页、欢迎页、错误页
- 多个应用共用同一套主框架，只调整品牌和少量宿主行为

---

## 2. 适合 page component 替换的部分

以下页面本身就是 router 级入口，天然适合通过页面注册替换：

- 登录页
- 欢迎页 / Dashboard
- 403 / 404 / 500
- 帮助页、关于页
- 任何结构、交互、数据来源差异明显的系统页

当前相关能力已经存在：

- contribution 类型支持 `builtinPages`
- 宿主启动时会注册 `builtinPages`
- router 再按 `componentId` 解析页面组件

因此，从机制上说：

- 登录页完全可以整页替换
- 欢迎页和系统页也完全可以整页替换

这类页面不必为了“配置化”而强行做成一堆零散字段。

---

## 3. 不应只靠 page 替换的部分

有一类内容并不属于某一个页面，而是整个 shell 的公共外观与元信息。这些内容如果只靠页面替换，会导致重复实现和配置分散。

典型包括：

- 应用名称、简称、logo、品牌标识
- 浏览器 `document.title`
- `favicon`
- 侧边栏品牌区
- 顶栏或移动端顶部品牌展示
- 帮助、关于、支持等应用级链接
- 默认首页路径、应用级外链

这些内容的共同特点是：

- 会在多个页面、多个布局区域重复出现
- 不属于单个页面自己的结构
- 换应用时几乎一定会变

因此更适合抽成 shell contribution 配置，而不是散落在多个页面组件里。

---

## 4. 登录页为什么建议保留两层能力

登录页是最容易产生分歧的地方。

### 4.1 什么时候直接替换整个登录页

如果某个应用存在以下诉求，直接替换 `LoginPage` 最合理：

- 布局完全不同
- 登录方式完全不同
- 需要额外步骤，例如租户选择、扫码、MFA、SSO 引导
- 视觉风格与默认登录页差异很大

### 4.2 什么时候只需要配置默认登录页

如果诉求只是：

- 换 logo
- 换标题、副标题
- 换 hero 文案
- 换背景图或品牌插画
- 控制是否显示 demo hint

那么为了这些变化去复制整页登录逻辑，成本偏高。默认登录页应支持读取 contribution 中的品牌与 UI 配置。

所以建议保留两层：

- 默认登录页支持 contribution 配置
- 特殊应用仍可通过 page component 完整替换

这不是二选一，而是“默认低成本 + 特殊高自由度”。

---

## 5. 建议的边界划分

### 5.1 page-level

继续通过 router + `builtinPages` 处理：

- `login`
- `dashboard`
- `forbidden`
- `notFound`
- `serverError`
- 其他系统页入口

适用原则：

- 页面结构变化大
- 页面逻辑需要完全自定义
- 页面需要独立维护自己的数据、交互和视觉

### 5.2 shell-level

建议通过 contribution 配置处理：

- 应用品牌信息
- 浏览器标题与 favicon
- 侧边栏 / 顶栏品牌展示
- 默认首页路径
- 帮助 / 关于 / 支持链接
- 默认登录页的品牌文案槽位

适用原则：

- 多处复用
- 不属于单个 page
- 主要是“换应用画像”而非“换页面结构”

---

## 6. 对当前实现的启示

当前 contribution 类型中已经有：

- `app.name`
- `app.shortName`
- `app.logoUrl`
- `app.defaultHomePath`
- `builtinPages`

但当前宿主实际使用得还不完整：

- `builtinPages` 已经进入了可替换链路
- `defaultHomePath` 已有明确场景
- `name`、`shortName`、`logoUrl` 还没有形成统一的 shell 运行时消费入口

这说明当前更需要补的是“shell 级配置消费”，而不是再发明一套新的页面替换机制。

优先级建议如下：

1. 建立统一的 shell branding/runtime config
2. 让 `AppBrand`、`document.title`、`favicon` 读取该配置
3. 默认登录页读取品牌文案与视觉配置
4. 对结构差异明显的场景，继续使用 `builtinPages` 替换整页

---

## 7. 推荐最小方案

如果只做最小可行改造，建议先覆盖以下能力：

- 品牌名
- 简称
- logo
- document title
- favicon
- 默认首页路径
- 默认登录页标题 / 副标题 / hero 文案

同时坚持以下边界：

- 登录页仍允许整页替换
- Dashboard / 403 / 404 / 500 仍以 `builtinPages` 为主
- 不把所有系统页都强行改成纯配置驱动

这样既能保持主框架复用，又不会把扩展模型做得过重。

---

## 8. 结论

主框架复用的关键不是“所有东西都配置化”，而是“把 page 替换和 shell 配置的边界划清楚”。

建议结论：

- 系统页：优先保留 page component 注册替换能力
- 壳层品牌与全局元信息：补充 contribution 配置能力
- 默认页面：尽量支持品牌配置，降低重复重写成本

最终形成的模型应是：

- 主框架负责稳定的布局、路由、权限、插件装载
- contribution 负责应用画像
- page component 替换负责少数结构差异极大的页面
