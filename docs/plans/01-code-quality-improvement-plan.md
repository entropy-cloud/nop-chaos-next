# 代码质量改进计划

## 1. 拟制原因

- 用户先提出“评估这个项目的代码质量”
- 在评估过程中，我对仓库进行了抽样阅读，并执行了 `pnpm lint`、`pnpm typecheck`、`pnpm test`
- 评估结果显示：项目整体架构分层较清晰，但工程质量门禁尚未闭合，主要问题集中在 `packages/ui` 的 lint 不通过、测试中的包导出与导入方式不一致，以及部分初始化与国际化细节仍需收口
- 用户随后明确补充了一条重要约束：`packages/ui` 包内部使用相对路径，工作区其他包和应用在消费 UI 能力时统一从 `@nop-chaos/ui` 集中导入
- 因此需要形成一份可执行的质量改进计划，用于后续分阶段治理，并将“UI 包内部相对路径、包外统一入口导入”作为明确规范写入计划

---

## 2. 当前质量结论

### 2.1 优点

- monorepo 分层清楚，`apps/main`、`packages/shared`、`packages/core`、`packages/plugin-bridge` 职责边界比较明确
- TypeScript 严格模式已开启，`pnpm typecheck` 当前可通过
- 共享逻辑、插件桥接、菜单校验、AMIS 适配等关键模块已有一定抽象
- 已具备 Vitest 与 Playwright 基础设施，不是完全缺失测试的仓库

### 2.2 主要问题

- `pnpm lint` 未通过，问题集中在 `packages/ui`
- `pnpm test` 未通过，暴露出 UI 包导入与导出策略不一致
- `packages/ui` 内部存在风格漂移，部分代码明显来自外部模板或生成代码，尚未完全纳入当前仓库规范
- 存在不一致的导入方式，与“UI 包内部相对路径、包外统一从 `@nop-chaos/ui` 导入”的要求冲突
- 个别模块使用文件级布尔变量控制初始化，降低了测试隔离性与可预测性
- 存在少量硬编码文案，国际化一致性不够完整

---

## 3. 明确治理原则

### 3.1 UI 导入原则

后续代码统一遵循以下规则：

- `packages/ui` 包内部复用自身组件、hooks、utils 时，一律使用相对路径
- `packages/ui` 包外的业务代码、工作区包代码在消费 UI 能力时，统一从 `@nop-chaos/ui` 集中导入
- 不再新增 `@nop-chaos/ui/components/...`、`@nop-chaos/ui/lib/...` 这类深层导入
- 若现有代码仍有不符合上述规则的导入方式，应纳入清理范围并逐步迁移

### 3.2 质量门禁原则

- 以 `pnpm lint && pnpm typecheck && pnpm test` 全绿作为基本门槛
- 不接受“类型通过但 lint 失败”或“lint 通过但测试失效”的半完成状态
- 对生成代码或外部迁移代码，要么完成本仓规范对齐，要么明确局部例外，不保持模糊状态

---

## 4. 分阶段改进计划

### Phase 1：恢复工程基线

目标：先把仓库恢复到可稳定验证的状态。

主要事项：

1. 将 `packages/ui` 内部导入统一收敛为相对路径
2. 清理工作区其他包和应用中对 `@nop-chaos/ui` 的深层导入，统一收敛到包入口
3. 处理 `packages/ui` 中阻塞 lint 的 `any` 与无用导入问题
4. 再次执行 `pnpm lint`、`pnpm typecheck`、`pnpm test`

完成标准：

- `pnpm lint` 通过
- `pnpm typecheck` 通过
- `pnpm test` 通过

### Phase 2：收口 UI 包规范

目标：让 `packages/ui` 真正符合仓库规范，而不是“能跑但噪音很多”。

主要事项：

1. 统一 `packages/ui` 的导入风格、引号风格和文件书写风格
2. 检查是否仍有从外部模板迁移留下的自引用路径、异常导出方式、未使用类型
3. 对必须保留的特殊模式进行最小化说明，避免后续反复引入相同问题
4. 补充 `packages/ui` 的最小测试覆盖，至少覆盖关键导出与基础组件装配

完成标准：

- UI 包不再成为 lint/test 的主要不稳定来源
- UI 包内部相对路径规则与包外统一入口导入规则都得到贯彻

### Phase 3：增强运行时健壮性

目标：降低初始化、副作用、插件运行时边界导致的不稳定性。

主要事项：

1. 评估并收敛 `apps/main/src/App.tsx`、`apps/main/src/hooks/useAuth.ts` 中的文件级初始化标记
2. 为插件加载、共享模块注册、鉴权 bootstrap 增补失败路径测试
3. 在 SystemJS 相关逻辑中补足更一致的运行时防御与错误提示
4. 审查全局状态与副作用注册的卸载逻辑，减少测试与热更新时的隐性状态污染

完成标准：

- 关键初始化链路更可测试
- 插件与鉴权相关错误更容易定位

### Phase 4：完善一致性与可维护性

目标：提升长期维护体验，而不是只修当前报错。

主要事项：

1. 清理硬编码文案，补齐国际化接入
2. 为菜单映射、路由解析、权限过滤等共享逻辑补充更完整测试
3. 将常见质量问题沉淀为开发规范文档或 PR 检查清单
4. 视情况补充 CI 工作流，自动执行质量门禁

完成标准：

- 关键共享逻辑有稳定回归保护
- 新增代码更难再次引入导入规则混乱、风格漂移和未验证提交

---

## 5. 优先级排序

### P0

- 禁止新增不符合规则的 UI 导入方式
- 修复现有 UI 内部自引用和包外深层导入导致的测试失败
- 修复 `packages/ui` 的 lint 阻塞项
- 恢复 `pnpm lint`、`pnpm typecheck`、`pnpm test` 全绿

### P1

- 收口 `packages/ui` 风格与导出规范
- 为插件加载、路由与鉴权补充更有针对性的单测
- 处理初始化哨兵变量带来的测试与运行时风险

### P2

- 完善 i18n 一致性
- 完善 CI 质量门禁与开发文档

---

## 6. 建议执行顺序

建议按以下顺序推进：

1. 全仓扫描并修正 UI 包内部相对路径与包外统一入口导入
2. 修复 `packages/ui` 中当前 lint 报错
3. 跑通全仓 lint、typecheck、test
4. 补关键回归测试
5. 再处理初始化模式、i18n 与规范沉淀

---

## 7. 验收方式

每一轮治理完成后，至少执行以下命令：

```bash
pnpm lint
pnpm typecheck
pnpm test
```

如果改动涉及插件加载、路由联动或宿主与插件桥接，补充执行：

```bash
pnpm test:e2e -- tests/e2e/plugin-demo.spec.ts
```

---

## 8. 预期结果

完成本计划后，项目应达到以下状态：

- `packages/ui` 内外导入边界清晰统一，不再出现内部自引用和包外深层导入混乱
- 质量门禁恢复可用，基础验证命令稳定通过
- `packages/ui` 从当前的主要噪音源转为稳定基础包
- 插件、路由、鉴权等关键链路的可维护性和可测试性提升
- 后续新增功能可以在更低回归风险下推进
