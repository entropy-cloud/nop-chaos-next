# 整体样式与交互规则

## 1. 目的

本文档用于约束宿主应用的整体视觉与交互规则，避免页面在迭代过程中出现样式分散、反馈不一致、桌面端与移动端行为割裂的问题。

适用范围：

- `apps/main`
- `packages/core`
- `packages/ui`

## 2. 交互反馈原则

### 2.1 可点击元素

可点击元素需要同时具备以下反馈中的至少两种：

- hover 背景或前景色变化
- `cursor: pointer`
- active / selected 状态
- focus-visible 可见焦点

适用对象：

- 侧栏菜单项
- 可点击表格行
- 卡片式列表项
- 自定义按钮容器

说明：

- 原生 `button` 和 `a` 在浏览器中不一定总是显示小手，但在后台壳子里，整行点击或自定义导航项建议显式补 `cursor-pointer`
- 纯展示信息块不应添加 pointer，避免误导

### 2.2 Hover 强度分级

统一将 hover 反馈分为三档：

- `subtle`：编辑型列表、表格子项、弱提示区域
- `default`：普通可感知 hover
- `interactive`：导航项、可点击行、重要列表项

交互密度越高，hover 越应该明确，但不能抢过 active 状态。

## 3. 侧栏规则

### 3.1 展开态

- 展开态直接展示树形结构
- 有子菜单的分组项和叶子项使用不同 hover 风格
- 当前激活项必须有稳定且持续的 active 状态，不能只靠 hover 区分

### 3.2 收起态

收起为图标栏后，必须补足信息可见性：

- 无子菜单项：hover 显示标题浮层
- 有子菜单项：hover 显示浮层子菜单
- 浮层中需要显示分组标题和子项列表
- 子项可直接点击跳转

说明：

- 收起态只显示图标时，不能要求用户记忆全部图标语义
- 桌面端可使用 hover 浮层；移动端不依赖 hover

### 3.3 鼠标样式

以下元素需要显式 `cursor-pointer`：

- 菜单项主点击区
- 浮层中的子菜单项
- 可点击表格行
- 卡片式可点击入口

以下元素不应使用 pointer：

- 纯文本标签
- 不可用状态
- 仅做容器布局的包裹层

## 4. 顶部导航规则

### 4.1 桌面端

- 使用顶部 tabs 导航条承载多标签页交互
- tabs 区只在大屏显示

### 4.2 移动端

- 不显示桌面 tabs 条
- 使用精简移动端 top bar
- 结构建议为：左侧菜单按钮 + 中间当前页标题 + 右侧用户入口

原因：

- 移动端横向空间不足，不适合长期展示 tabs
- 用户最需要的是主导航入口和当前上下文信息

## 5. 表格规则

### 5.1 表格行交互约定

表格行统一使用 `@nop-chaos/ui` 中的约定函数：

- `getTableRowClassName('default')`
- `getTableRowClassName('interactive')`
- `getTableRowClassName('subtle')`

推荐使用方式：

- 普通展示表格：默认 `TableRow`
- 整行可点击：`interactive`
- 编辑型子表或弱反馈区域：`subtle`

### 5.2 状态叠加

当表格行还有 warning、dirty、selected 等状态时：

- 状态背景优先表达状态
- hover 作为补充，不应完全覆盖状态语义

## 6. 第三方样式边界

第三方库的全局工具类和宿主样式体系不能混用。

规则：

- 优先使用宿主应用自己的 token 和 utility
- 避免引入会污染全局类名空间的第三方 helper 样式
- 如必须引入，应优先考虑隔离范围，而不是直接全局注入

相关案例：

- `docs/12-amis-helper-css-conflict.md`

## 7. 落地建议

新增页面或组件时，优先检查：

- 是否复用了现有 UI 约定
- hover / active / focus 是否完整
- 桌面端与移动端是否分别定义了合理行为
- 是否错误依赖了第三方全局样式

如果同类组件已经存在实现，优先抽取到 `packages/ui` 或 `packages/core`，不要在页面里重复发明一套新的交互规则。
